Viewport as AOI

Standard remote-sensing tools ask users to define an area of interest before returning data — draw a polygon, upload a shapefile, or enter bounding-box coordinates. This proof of concept tests a simpler premise: a user who has navigated a map to an area they care about has already implicitly defined their AOI. Formalising that selection adds friction without adding analytical clarity for a first-pass inspection.

Hypothesis

The current map viewport is a sufficient area of interest for vegetation-index inspection at typical inspection zoom levels. The user's navigation act — panning to a farm, a watershed, a landscape of interest — is itself an implicit AOI declaration. Capturing that implicitly, rather than requiring an explicit drawing step, reduces the interaction cost to near zero.

The test: build a browser tool that returns an NDVI surface for whatever the user currently sees on the map, with no drawing, uploading, or coordinate entry required. If the result is analytically useful and the interaction feels natural, the hypothesis holds. If the viewport proves too coarse or too variable to be useful, the hypothesis fails and polygon-drawing becomes justified.

What the demo shows

The demo renders an NDVI surface as a Plotly.js 3D visualisation for the visible map viewport. The surface height represents the NDVI value — not elevation. Contour lines mark NDVI isolines across the grid. A MapLibre map with a MapTiler basemap provides spatial context.

In the production pipeline, the surface would be derived from Sentinel-2 L2A satellite imagery via the Sentinel Hub Process API, routed through a Cloudflare Worker that handles OAuth credentials securely. In this proof of concept, the surface is generated from a procedural fixture — a spatially coherent synthetic NDVI grid seeded by the viewport bounds and selected date. The interaction model is identical to the live pipeline.

This demo runs on synthetic fixture data. No external API calls are made and no satellite credentials are required. The fixture produces plausible spatial variation across the viewport — it is intended to demonstrate the interaction model, not to represent real vegetation conditions.

The live data pipeline — Cloudflare Worker proxy, Sentinel Hub Catalog and Process APIs, real Sentinel-2 L2A imagery — is designed and documented but not yet connected.

Design decisions

Several decisions were made deliberately to keep the interaction model minimal.

The AOI is always the current viewport. There is no draw tool, no saved area, and no coordinate input. If a user wants a different area they move the map. This removes an entire class of UX complexity while preserving the core analytical value: spatial vegetation-index inspection.

The index selection is fixed to NDVI for the first pass. Additional indices — NDRE, NDWI, EVI — are straightforward additions once the pipeline is live, but adding them before the interaction model is validated introduces unnecessary optionality.

The 3D surface uses NDVI as the z-axis height, not elevation. This is an analytical surface, not a terrain visualisation. The label is explicit: z = NDVI. Elevation-height mode using a DEM grid is a defined future enhancement.

Technical stack

MapLibre GL JS — interactive map rendering MapTiler Cloud — basemap and terrain tile provider Plotly.js — 3D analytical surface visualisation Sentinel-2 L2A — satellite imagery source (production path) Sentinel Hub Process API — RGB and index products (production path) Sentinel Hub Catalog API — scene search and selection (production path) Cloudflare Workers — secure OAuth proxy (production path) GitHub Pages — static frontend hosting
