Ambiguous Documents

Product work usually enters the organisation as documentation before it enters the codebase. The danger is not that early documents are ambiguous; the danger is pretending they are already precise enough to hand over.

This module treats a brief as a documentation system made of eight working documents. They should be ordered so ambiguity decreases as you read down: the top keeps the question open, the middle narrows the design surface, and the bottom is where engineering can build without inventing missing decisions.

The documents are provisional, not sacred. They should reduce ambiguity enough for the next responsible move, while staying open to evidence that changes the bet. When documentation cannot absorb feedback, the organisation accumulates Feedback Debt.

What the documents are for

Early documentation is the artefact before convergence. It holds the problem open so product, design, and engineering can explore without mistaking a first description for a final decision. A PRD — or whatever the organisation calls its closing document — comes later, once the solution is understood.

Writing the closing document before discovery is the common failure mode. It looks decisive, but usually means the documentation stopped holding ambiguity before the team had enough signal.

There are two common documentation paths. Improvement work changes something that already exists; it needs a Where We Are document as an anchor, because design and engineering are not starting from zero. New-feature work has no such ground to stand on, so the People document has to carry more weight — it is the only scaffolding available.

The eight documents

Each document does one job. The further down the set you go, the less room there should be for interpretation.

1 · Problem Document

Why this work exists at all. The point of departure. Must be agreed before anything else moves. Ambiguity is highest here and that is correct — premature precision at the top of the document set is how solutions get smuggled in as problems.

2 · Direction Document

The desired change in the world. Outcome, not solution. This is the earliest document where design can start diverging, and the last document where the work is still fully open. After this point the shape of the thing begins to matter.

3 · Where We Are Document (improvement work only)

A shared baseline — how the current thing works, what is broken, what the constraints of the existing system are. Without this document, the first week of any improvement project is spent reconstructing it anyway.

4 · People Document

Who is affected, what they are trying to accomplish, under what conditions. Load-bearing in new-feature documentation where there is no existing feature to orient around. In improvement work this still matters, but Where We Are shares the weight.

5 · Scope Document

In, out, and open questions. The open-questions column is the one to watch: it should shrink as discovery progresses. A scope document that stays static is a scope document that is not being challenged.

6 · Slice Document (stories)

User-facing units of value. Written loosely here to keep exploration possible; tightened into ticket-ready form later, in the PRD or directly in the backlog. If you cannot yet write this document, the direction above probably is not specific enough.

7 · Constraints Document

The boundaries the solution must operate within: legal, technical, deadlines, non-functional needs, integration dependencies. Should be short. If this document gets long, scope decisions have crept in disguised as constraints.

8 · Acceptance Criteria Document

Binary pass/fail conditions. The most precise document in the set, and the one with the least ambiguity. If these are hard to write, the documents above are not yet clear enough — which is information, not failure.

Three thresholds

A documentation set becomes ready in stages. Different kinds of work can start when different documents are stable enough. Treating those thresholds as one moment is how design starts too late or locks down too early.

Explore opens (after Direction)

The Problem and Direction documents together are enough to start exploration. You do not need users, scope, or stories yet. Waiting for them is how design shows up after decisions are already made.

Solution can commit (after Slices)

People, Scope, and Slice documents are all known. This is the right threshold for solution work to commit. Earlier than this, detailed work is often reworked. Much later than this, keeping everything open creates churn.

Dev can build (after Acceptance Criteria)

Fully unlocked at Acceptance Criteria. Earlier documents contribute gradually; the bottom of the set is where dev can build without interpretation.

Zoom out

This documentation problem happens many times over in any organisation. 04 · Learning & Feedback asks where an organisation should commit to knowing — which paths it can afford to run sequentially and which must stay genuinely open. This module shows how one path is documented while it runs.

The same bet, at two altitudes. A document set that pretends to be final is a portfolio that pretends complexity is manageable — the same failure mode, one level down.
