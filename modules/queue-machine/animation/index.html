Flow Animation

Inspired by Henrik Kniberg's resource utilization trap: when management optimizes for busy people, the work product can move slower. The counter-intuitive move is to tolerate visible slack so value can flow. This version uses the same bursty arrivals and variable task sizes in both lanes; only the work-release policy changes.

Lesson Watch the work not the worker

Same incoming work, same station sequence, different policy. The busy lane releases up to nine items into the system and pays a congestion penalty when queues build. The flow lane limits WIP to three, so it sometimes looks idle while protecting throughput and lead time.

What the animation shows

The upper lane is optimized for resource utilization: incoming work is pushed into the system until the WIP limit is high. Because many items wait at handoffs, each station pays a small congestion and switching penalty. The lower lane is optimized for flow: it uses the same arrivals and task sizes, but releases less work at once.

The point is not that utilization is irrelevant. The point is ordering. First watch the work product: finished value, throughput, waiting, WIP, and lead time. Then use capacity deliberately. If the primary target is making every person look busy, the system can reward motion while punishing flow.

  1. Arrivals: work enters in bursts, so the average arrival rate hides local spikes.
  2. Size variation: each item has different work at stations A, B, and C.
  3. Push: the busy policy allows high WIP, which raises waiting and congestion.
  4. Pull: the flow policy caps WIP, accepting visible slack to reduce handoff queues.
  5. Measure: compare throughput, utilization, WIP, and waiting instead of only who looks busy.
